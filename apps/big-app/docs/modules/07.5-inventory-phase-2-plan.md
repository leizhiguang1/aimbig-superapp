# Inventory — Phase 2 plan

> **Status:** plan only, not yet built.
> Companion to [07-inventory.md](07-inventory.md).
> Drafted 2026-05-06 after the v1 inventory module was closed out.

This file scopes the two Phase-2 follow-ups the user explicitly wants
queued next:

1. **Per-outlet stock** — split the single global `inventory_items.stock`
   into a per-outlet quantity, unblocking POs / Stock Request / Transfer
   Orders / Returned Stock.
2. **`discount_cap` enforcement in `collect_appointment_payment` /
   `collect_walkin_sale`** — the column is set on items today but the
   sales RPCs don't check it.

Everything else from the deferred list (Coverage Payor, Loyalty BP,
vaccinations, prescriber-role gating, e-Invoice classification, barcode
uniqueness) is out of scope for this doc.

## 1. Per-outlet stock

> **Status:** SHIPPED 2026-05-06. Implementation diverged slightly from
> the original 9-step plan; recap below.
>
> - `inventory_item_outlets(item_id, outlet_id, …)` junction created with
>   per-outlet `cost_price`, `selling_price`, `stock`, `in_transit`,
>   `locked`, `stock_alert_count`, `minimum_stock_level`, `location`,
>   `is_sellable` — *all* the prototype's per-outlet attributes in one
>   table, not just stock.
> - `inventory_items.{cost_price,selling_price,stock,…}` kept as a
>   "default template" used when seeding outlet rows for new items or
>   new outlets — easier than dropping them and rewriting downstream
>   reads piecewise.
> - `inventory_movements.outlet_id` added + backfilled. RPCs (collect_*,
>   void_sales_order) all rewritten to use the junction.
> - `recordStockMovement(ctx, itemId, outletId, input)` enforces outlet
>   scope. The Stock Adjustment dialog renders an outlet picker when
>   more than one exists.
> - Item form gains an "Outlets" section that mirrors the kumodent
>   prototype: per-outlet pricing/alert/location/sellable plus an
>   "Apply above prices, stocks to all outlets" master toggle.
> - Items table renders the active outlet's stock cell (resolved from
>   `params.outlet` → outlet UUID).
> - Dashboard `LowStockCard` reads per-outlet `stock_status` from
>   `inventory_item_outlets`.
> - Auto-seed triggers ensure new items / new outlets get junction rows
>   automatically, no manual intervention needed.
>
> Full migration list and implementation notes in
> [07-inventory.md §Per-outlet split](07-inventory.md#per-outlet-split-shipped-2026-05-06).

### Why this is the keystone

Four full inventory tabs are gated on it:

- **Purchase Orders** — receiving has to land at *one* outlet
- **Stock Request** — outlet → HQ replenishment
- **Transfer Orders** — outlet → outlet rebalance
- **Returned Stock** — supplier returns from a specific outlet

It also unblocks per-outlet pricing and per-outlet location later —
those are simpler add-ons once the stock split exists.

### Recommended shape (decision)

A **dedicated `inventory_stocks` junction table is preferable to widening
`inventory_items` with one row per outlet**, because:

- Outlets come and go; items don't. Adding/removing an outlet shouldn't
  require a schema migration on the catalog.
- Most reads are per-item-per-outlet; a junction with composite PK gives
  us a single index (`item_id, outlet_id`) that covers both lookups.
- Movements naturally extend to per-outlet too — adding `outlet_id` to
  `inventory_movements` is a one-column migration, no shape change.

```sql
create table public.inventory_stocks (
  item_id      uuid not null references inventory_items(id) on delete restrict,
  outlet_id    uuid not null references outlets(id)         on delete restrict,
  stock        numeric(12,2) not null default 0,
  in_transit   numeric(12,2) not null default 0 check (in_transit >= 0),
  locked       numeric(12,2) not null default 0 check (locked     >= 0),
  stock_alert_count numeric(12,2) not null default 0,
  stock_status text generated always as (
    case
      when stock <= 0                  then 'out'
      when stock <= stock_alert_count  then 'low'
      else 'normal'
    end
  ) stored,
  updated_at   timestamptz not null default now(),
  primary key (item_id, outlet_id)
);
create index inventory_stocks_outlet_idx on public.inventory_stocks(outlet_id);
```

### Migration / cutover plan

1. **Create the table** (one migration).
2. **Backfill** — for every existing `inventory_items` row, insert one
   `inventory_stocks` row per active outlet under the same brand. Most
   sensible default: dump the entire global `stock` into the brand's
   *primary* outlet (or the outlet flagged on `outlets.is_primary` if
   we add that flag — otherwise pick the lowest-`code` outlet) and zero
   for the rest. Stop-the-world acceptable: the user has one outlet per
   brand today, so this is a no-op in practice.
3. **Add `outlet_id` to `inventory_movements`** — `nullable` initially,
   default-fill historical rows to the same primary outlet as step 2.
   Then `set not null` in a follow-up migration after we've verified
   nothing slipped through.
4. **Update RPCs** — `collect_appointment_payment` and
   `collect_walkin_sale` currently mutate `inventory_items.stock`.
   Switch to `update inventory_stocks set stock = stock - v_qty where
   item_id = ? and outlet_id = v_outlet_id`. The outlet is already on
   the parent appointment / sales order. `void_sales_order` walks the
   ledger same as today but adjusts `inventory_stocks` instead.
5. **Update `recordStockMovement` service** — accept an `outletId`
   parameter; UI passes the active-outlet context. Form-level manual
   stock edits (`updateInventoryItem`) — drop the `stock` field from
   the item form (it's ambiguous once stock is per-outlet) and force
   all manual edits through the Stock Adjustment dialog, which already
   has clear semantics.
6. **Drop or convert `inventory_items.stock` to a sum-view.** The
   doc currently flags this as a Phase 2 decision; recommend
   *converting it* to a generated column (`stock = sum over
   inventory_stocks`) so existing reads keep working. Cheaper than
   chasing every callsite.
7. **Stock Details dialog** — add an outlet filter pill row above the
   ledger; default to the active outlet. Movements query gains
   `outlet_id = ?` filter.
8. **Items table** — `stock_status` becomes per-outlet. Show the active
   outlet's stock in the row, with a tooltip listing every outlet's
   on-hand. Or simpler: add an "Outlet" column that defaults to the
   active outlet from the URL.
9. **Dashboard low-stock card** — already brand-scoped, becomes
   outlet-scoped automatically by reading `inventory_stocks` filtered
   by `ctx.outletId`.

### Risk

- The Collect Payment RPC rewrite is the only meaningful surface — it's
  already covered by `lib/services/__tests__/sales.test.ts`, so the
  rollback path is the test. Add per-outlet variants of the existing
  cases.
- Backfill is the cutover risk. Run it inside a transaction with a
  `select count(*) from inventory_items` vs `select count(*) from
  inventory_stocks` assertion at the end.

### Out of scope (still deferred even after this lands)

- **Per-outlet pricing** — separate `inventory_item_outlet_pricing`
  table. Easy to add once `inventory_stocks` exists; deliberately
  excluded here so the stock migration stays small.
- **PO / Stock Request / Transfer Orders / Returned Stock UI** — these
  are unblocked by the stock split but are their own modules. Build one
  at a time after the split lands.

## 2. `discount_cap` enforcement in collectPayment RPCs

> **Status:** SHIPPED 2026-05-06. Kept here for context; actual approach
> taken differs slightly from the plan below.
>
> - Service guard `assertLineDiscountCaps` extended to load
>   `inventory_items.discount_cap` alongside `services.discount_cap`.
> - UI `capFor(line)` in both sales paths
>   ([NewSaleDialog](../../components/sales/NewSaleDialog.tsx) and
>   [useBillingLines](../../components/appointments/detail/collect-payment/use-billing-lines.ts))
>   resolves the cap from whichever vocabulary applies.
> - **Defense-in-depth:** instead of editing the 700-line collectPayment
>   RPCs, a `BEFORE INSERT` trigger `sale_items_enforce_discount_cap`
>   on `public.sale_items` enforces the cap at the table level. Catches
>   every non-service caller automatically.

### Today (pre-2026-05-06, kept for posterity)

- `inventory_items.discount_cap` (numeric 5,2, 0–100) exists, settable
  via the item form.
- `services.discount_cap` exists and **is** enforced — see
  `assertLineDiscountCaps` in [lib/services/sales.ts](../../lib/services/sales.ts)
  and the matching server-side check inside
  `collect_appointment_payment`. The UI's `capPct` + on-blur clamp uses
  `services.discount_cap`.
- `inventory_items.discount_cap` is **not** read anywhere on the sales
  path. Both RPCs and the service-layer guard ignore it.

### Plan

Mirror the existing `services.discount_cap` enforcement, applied to sale
items whose `inventory_item_id` is set:

1. **Service-layer guard** — extend `assertLineDiscountCaps` in
   `lib/services/sales.ts`. Today it only loads service caps; widen the
   load to also fetch `inventory_items.discount_cap` for any line where
   `item_type !== 'service'` and `inventory_item_id` is set, then run
   the same `lineDiscount > capPct * lineGross` check. Same error shape
   (`ValidationError("'<item>' discount exceeds N% cap (RM X)")`).
2. **RPC mirror** — replicate the same check inside
   `collect_appointment_payment` and `collect_walkin_sale` so a non-UI
   client can't bypass it. A `select discount_cap from inventory_items
   where id = v_inv_item_id` per item is fine (catalog is small);
   compare to `v_line.discount / (v_qty * v_unit_price)` and `raise
   exception` if over.
3. **UI** — wire a `capPct` resolver inside `NewSaleDialog` /
   `BillingItemPickerDialog` for product/medication lines, identical
   to the service path. The "apply max" button + on-blur clamp pattern
   is already extracted, so this is a small delta.
4. **Tests** — add a `discount_cap` case to
   `lib/services/__tests__/sales.test.ts` that fails when an item-level
   cap is exceeded, mirroring the existing service-cap test.

### Why it can land independently of per-outlet stock

The cap is a per-item attribute, has no outlet dimension, and the data
already exists. This is essentially a 50-line change across the service
+ RPC + UI; safe to ship before the stock split.

### Risk

Low. The existing service-cap pattern is the template; failure mode is
"discount rejected when staff applied an over-cap discount" — exactly
the intent.

## Suggested order

1. **`discount_cap` enforcement** — small, independent, validates the
   pattern still works.
2. **Per-outlet stock keystone** — biggest single piece.
3. **Per-outlet pricing** — small follow-up to (2).
4. **POs → Stock Request → Transfer Orders → Returned Stock**, one tab
   at a time. Each is a self-contained module on top of (2).

## Open questions

- **Active-outlet source** — `ctx.outletId` exists already (URL-driven
  via `/o/[outlet]/...`). Confirm that's the source of truth before
  threading it through the RPCs, or do we want a UI-level outlet
  switcher inside the inventory page that's separate from the route?
- **Manual stock edit on the item form** — kill the `stock` field from
  the form once stock is per-outlet (forcing the Stock Adjustment
  dialog), or keep it as a "set initial stock for active outlet"
  shorthand? Recommend kill-it; it's ambiguous once outlets exist.
- **`stock_alert_count`** — currently global per item; should this also
  go per-outlet? Recommend yes (carry on the junction table) since a
  busy outlet may need a higher alert threshold than a slow one. Same
  cost.
